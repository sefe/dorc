# HLPS — Request Grid Substring-Scan Performance Fix

**Status:** IN REVIEW
**Date:** 2026-05-05
**Author:** Ben Hegarty (with Claude Opus 4.7)
**Topic slug:** `request-grid-perf`

---

## 1. Problem Statement

On 2026-05-05 the production DOrc API host (GMPR-DORAPP02) and the SQL Server backing it experienced sustained CPU saturation. A diagnostic taken during the incident identified the dominant CPU consumer as a single query pattern: a `SELECT … FROM [deploy].[DeploymentRequest]` (mislabelled as `DeploymentResult` by the SSMS Copilot session, but the projected columns — `Id, Components, RequestedTime, StartedTime, CompletedTime, Status, Log, IsProd, Project, …` — are unambiguously `DeploymentRequest`) executed against a parameter typed `nvarchar(4000)` and matched with `LIKE '%@p_1_contains%'`. The diagnostic reported ~330–375K logical reads per query, ~166 active workers across 8 schedulers, and ~18 concurrent identical sessions — a fully saturated 8-CPU box.

The query is generated by the DOrc API endpoint that backs the request-status grid in the dorc-web UI (`RequestsStatusPersistentSource.GetRequestStatusesByPage`). It funnels the user-selected grid filters through a centralised helper, `DataPagerExtension.ContainsExpression`, which builds a LINQ expression `entity.Property.Contains(value)` for any string column. EF Core compiles `string.Contains` into a non-SARGable `WHERE [Col] LIKE '%' + @p + '%'` predicate, forcing a full scan on every filtered request.

Two concrete code-level defects make the cost particularly acute on the request-status page:

1. **`EnvironmentNameExact` filter routes through `ContainsExpression`** (`RequestsStatusPersistentSource.cs:60-66`, introduced in commit `d9167efe` "adapted page-monitor to make env specific monitor page"). The path name encodes the intent ("Exact"), and the comment on the surrounding block says the filter exists for "the env specific monitor". The env-pinned dashboards in the dorc-web frontend always know the full environment name and want exact-match semantics, but the helper still emits a substring scan. This filter is one of the most frequently invoked predicates in production because every open env-monitor session polls the endpoint.

2. **PR #338 (`f91dfc6b`, 2026-04-10) tripled the substring-scan frequency per page request on the global request grid.** That PR replaced the single Details filter input on `page-monitor-requests.ts` with three independent inputs (Project, EnvironmentName, BuildNumber) and removed the multi-sort guard *in that component's dataProvider only*. Each non-empty filter input now produces its own substring scan; when the three values are distinct they are ANDed together (`hasDistinctDetailValues == true`), otherwise they are ORed via `WhereAny` (`hasDistinctDetailValues == false`). Per-request cost on `page-monitor-requests.ts` is now up to three substring scans instead of one. Note: `env-monitor.ts` was *not* changed by PR #338 — it retains its multi-sort guard and still pushes a single `detailsFilter` value to both `Project` and `BuildNumber` paths simultaneously (`env-monitor.ts:199-205`), which always lands on the OR-path. The two components therefore have materially different filter-shape behaviour and the IS must address both.

PR #338 has been in production for 25 days, so it is not a sole proximate trigger for the 2026-05-05 spike. The most plausible trigger sequence is: PR #338 increased steady-state cost; daily growth in `deploy.DeploymentRequest` raised per-scan logical reads beyond a CPU-saturation threshold; an event on 2026-05-05 (a stats refresh-induced plan flip and/or a workload concurrency spike — Session 63 was observed running `UPDATE STATISTICS` on the same table during the incident) tipped the system from "expensive but tolerated" to "saturated". The trigger event is operational; the fix this HLPS addresses is structural.

## 2. Goal

Eliminate the structural conditions that cause the request-status grid to issue non-SARGable `LIKE '%…%'` scans against `deploy.DeploymentRequest`, so that the pre-incident workload (and reasonable growth on top of it) does not re-saturate the SQL host. Restore the request-status page to a sub-second, index-driven access pattern under representative production data and concurrency.

## 3. Scope

### In Scope
- Correct the `EnvironmentNameExact` filter so it generates an equality predicate and uses an existing or newly-added index on `deploy.DeploymentRequest(Environment, …)`.
- For the per-field grid filters introduced in PR #338 (Project, EnvironmentName, BuildNumber), choose and implement a SARGable predicate strategy (default candidate: `StartsWith`) consistent with the actual UI/UX requirement. The chosen predicate semantics must apply consistently across both call paths in `RequestsStatusPersistentSource.GetRequestStatusesByPage`: the AND-path (`hasDistinctDetailValues == true`, used by `page-monitor-requests.ts`) and the OR-path (`hasDistinctDetailValues == false`, used by `env-monitor.ts`'s shared `detailsFilter`). The IS must specify behaviour for both paths.
- Add the supporting non-clustered index(es) on `deploy.DeploymentRequest` required to make the chosen predicates index-seekable in production volumes. The IS must consider plan-choice interaction with the existing `IX_Status_IsProd` index for queries that filter by Status and Environment together (the env-monitor case).
- Verify that the change holds under representative load and that no other consumer of the affected code path regresses (Performance test plan — see §6).
- Audit other callers of `DataPagerExtension.ContainsExpression` (Database, ManageProjects, PropertyValuesAudit, PropertyValues, DaemonAudit, DaemonObservation) to confirm whether any are at similar risk and document findings. **Definition of "critical instance":** a caller is critical (and pulled into this HLPS's scope) if all three of (a) the underlying table has > ~100K rows in production, (b) the filter is invoked from a UI surface that is polled or auto-refreshed (not user-triggered one-shot queries), and (c) at least one filter column lacks a supporting index. Otherwise the audit finding is recorded for follow-up but does not block this HLPS.

### Out of Scope
- A general-purpose refactor of `DataPagerExtension.ContainsExpression` itself. The helper has many call sites; this HLPS does not propose a project-wide change to its semantics.
- Operational mitigations (killing runaway sessions, polling-cadence changes in clients, SQL plan-pinning hints, SQL-side index maintenance jobs). These are owned by the operations / DBA function and are out-of-band of this code change.
- Changes to other paged-grid endpoints (servers, environments, scripts, audit) unless the audit in §3-In-Scope identifies a critical instance.
- Schema changes to `deploy.DeploymentRequest` beyond non-clustered index additions.
- Migration to a different ORM, EF version upgrade, or SQL-side stored-procedure replacement of the grid query.

## 4. Constraints

- **Backwards compatibility of the request-status API contract.** Existing API consumers (the dorc-web frontend at minimum) must continue to function. If the filter semantics change (substring → prefix), the change must be made explicit across both backend and frontend, with no silent behaviour shift. *No silent shift* means at minimum: (a) user-facing release notes in the next release bundle; (b) updated input affordance in the affected dorc-web components — placeholder text, label, or tooltip — to reflect the new semantic. The IS specifies which affordance is used.
- **Production-safe schema changes.** Any new index on `deploy.DeploymentRequest` must be deployable against a populated production database without an exclusive lock that would block the application. Online index creation (Enterprise) or a documented maintenance-window approach must be specified.
- **Existing migration pattern must be followed.** Schema changes ship as a versioned migration script consistent with the rest of the DOrc codebase.
- **EF Core version is fixed.** No EF Core upgrade as part of this work.
- **Cognitive complexity must not increase.** Per CLAUDE.md, the fix favours low-risk, low-complexity solutions. Replacing one bad LINQ expression with a clearer one is preferred over introducing a query-builder abstraction.
- **No regression in existing tests.** All current tests must continue to pass; new tests cover the changed predicates.

## 5. Success Criteria

1. The `EnvironmentNameExact` filter, when invoked from the env-pinned monitor page (`env-monitor.ts`), generates `WHERE [Environment] = @p` (equality) and the resulting query plan uses an index seek (not a scan) on `deploy.DeploymentRequest`.
2. The per-field UI filters from PR #338 generate the agreed predicates per U1, in **both** the AND-path and the OR-path of `GetRequestStatusesByPage`: `Project` → `StartsWith` (`LIKE '@p%'`, SARGable, index seek). `EnvironmentName` → `StartsWith` (SARGable, index seek). `BuildNumber` → substring retained (`LIKE '%@p%'`). When the user provides a Project or Environment filter (SARGable seek), the BuildNumber predicate operates on the row set already narrowed by the Project / Environment seek and the residual scan is bounded. **BuildNumber-only filtering** (no Project, Environment, or other indexed predicate) is explicitly out-of-scope of the perf guarantee in this HLPS — the IS may either accept the residual scan as a known limitation or document a deferred follow-up.
3. Logical reads per page query, measured against a representative `deploy.DeploymentRequest` (matching production row counts and per-environment skew within an order of magnitude), drop materially below the observed ~370K range for both the env-monitor filter combinations and for typical multi-filter grid combinations. The exact ceiling is set in the JIT Spec for the perf step; the HLPS's role is to require that the metric is captured and meets a justifiable seek-driven target.
4. *(Operations-owned, non-blocking for delivery sign-off — see §6.)* CPU on the SQL host does not exceed pre-#338 baseline under the workload pattern that triggered the 2026-05-05 incident, when replayed against the post-fix code.
5. No correctness regression: every row that the old `Contains`-based predicate matched against representative production data is still matched in the BuildNumber case. Where the semantic is intentionally tightened (Project / EnvironmentName: substring → prefix), the change is (a) listed in user-facing release notes and (b) reflected in a visible UI affordance change in the affected dorc-web components per Constraint §4.
6. The audit in §3 produces a written list of other `ContainsExpression` consumers and flags any meeting the §3 "critical instance" definition; that list is recorded in this HLPS folder for follow-up.
7. New index(es) ship through the existing SSDT database project (`src/Dorc.Database`). The IS chooses between the two patterns observed in the codebase — separate file under `Schema Objects/Schemas/deploy/Tables/Indexes/` (used by most tables), or inline `CREATE NONCLUSTERED INDEX` in the `.table.sql` (the existing pattern for `DeploymentRequest`). Deployment must not require a blocking exclusive lock on production (subject to U4 resolution on edition).

## 6. Verification Approach (high-level)

- **Unit / integration tests** verify that the new predicates produce equality / `StartsWith` SQL (assertable via EF query string capture) for both the AND-path and the OR-path. Existing grid filter tests continue to pass.
- **Local performance check** against a seeded `deploy.DeploymentRequest` of representative size verifies index seek usage in the actual plan and captures the logical-read count for SC3.
- **Production verification** is owned by the operations team; SC4 is an operations-owned post-deploy criterion, not a delivery gate. No code change in this HLPS is gated on a production replay; we deliver a defensible structural fix and the operations team owns the post-deploy validation.

## 8. Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Behaviour change from substring → prefix is user-visible. Users may rely on substring matching as a daily workflow. | MEDIUM | U1 is resolved (`StartsWith` for Project / EnvironmentName; substring for BuildNumber). The substantive mitigation is now SC5 + Constraint §4: visible UI affordance change (placeholder/tooltip) plus release notes. If post-deploy feedback reveals users genuinely needed substring on Project/Environment, follow-up HLPS would specify an indexed-substring approach (full-text or computed column). |
| Adding an NC index on a hot, populated production table can briefly affect inserts. | LOW–MEDIUM | Online index build (Enterprise) or scheduled maintenance window (Standard). U4 disambiguates per-step before this work enters Delivery. |
| New index(es) may interact poorly with the existing `IX_Status_IsProd` for queries that filter on both Status and Environment, leading to a different plan regression (e.g., the env-monitor query that combines `Status = …` and `Environment = …`). | MEDIUM | The IS perf step must capture the actual plan for the env-monitor query under the post-fix indexes, not just the isolated Environment-filter case. If a regression is observed, the IS specifies a covering or composite index that handles both predicates jointly. |
| The audit in §3 may surface another caller of `ContainsExpression` that meets the "critical instance" definition and arguably belongs in this HLPS scope. | LOW | If it does, escalate to user; either expand scope explicitly or open a follow-up HLPS. Default is to keep this HLPS narrow. |
| The 2026-05-05 trigger event is unconfirmed. If the trigger was a one-off (e.g., a rogue user opening 50 dashboards), the structural fix is right but the incident may recur from a different trigger. | LOW | Out-of-band: the operations team is responsible for trigger investigation. This HLPS is structurally complete regardless. |
