## 1. Overview

S-006 is the formal performance-verification step for the request-grid fix. It produces measured evidence that the structural change in S-002 / S-003 / S-004 delivers the cost reduction HLPS SC3 requires.

This step **cannot be executed from the dev workstation alone**: it requires a SQL Server instance with a representative seeded `deploy.DeploymentRequest` table (production-like row counts and per-environment skew within an order of magnitude — HLPS SC3) plus the application running against it. The runbook below is what the user / DBA executes in the appropriate environment.

The S-006 deliverable is this runbook **plus** the captured numbers (added to a §6 "Results" section once the runbook has been executed). Until §6 is filled in, S-006 is incomplete.

## 2. Prerequisites

- **Branch:** `perf/request-grid-sargable-filters` deployed to a non-production environment with realistic data. The branch contains:
  - S-002 indexes shipped via the SSDT project (`IX_DeploymentRequest_Environment`, `IX_DeploymentRequest_Project`).
  - S-003 `EnvironmentNameExact` equality predicate.
  - S-004 `Project` / `EnvironmentName` `StartsWith` predicates in both AND-path and OR-path.
- **Baseline:** a comparable environment running `main` (or a snapshot of `main` taken before this branch was merged) for pre-fix capture. If no separate environment is available, the DBA can capture pre-fix numbers by `DROP INDEX` + reverting the predicates temporarily, capturing, then restoring — but a dedicated baseline is preferred.
- **Seeded data:** `deploy.DeploymentRequest` populated to within an order of magnitude of production row count (typically the non-production env restored from a recent prod backup will satisfy this). Per-environment skew should approximately match production.

## 3. Queries to measure

Each query is exercised twice: once against the pre-fix baseline, once against the post-fix branch. For each capture:

- **Logical reads** — `SET STATISTICS IO ON;`
- **Plan shape** — actual execution plan (`SET STATISTICS XML ON;` or SSMS "Include Actual Execution Plan").
- **Elapsed time** — `SET STATISTICS TIME ON;`

### Q-A: env-monitor `EnvironmentNameExact` (alone)

```sql
-- Replace @env with a realistic non-empty environment name from your seed.
DECLARE @env NVARCHAR(64) = 'PROD-NA';

SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- The actual query is generated by GetRequestStatusesByPage; the equivalent
-- T-SQL the API issues is roughly:
SELECT TOP (50) [d].[Id], [d].[Components], [d].[RequestedTime], [d].[StartedTime],
       [d].[CompletedTime], [d].[Status], [d].[Log], [d].[IsProd], [d].[Project],
       [d].[Environment], [d].[BuildNumber], [d].[UserName], [d].[RequestDetails],
       [d].[UncLogPath], [d].[CancelledBy], [d].[CancelledTime]
FROM [deploy].[DeploymentRequest] AS [d]
WHERE [d].[Environment] = @env  -- pre-fix: LIKE '%' + @env + '%'
ORDER BY [d].[Id] DESC;

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

**Expected post-fix:** index seek on `IX_DeploymentRequest_Environment`. Logical reads bounded by the per-environment row count + a few index pages. For an environment with 10K matching rows out of 1M total: roughly 100–500 logical reads.

**Pre-fix:** clustered-index scan on `PK_DeploymentRequest`. Logical reads bounded by the full table. ~370K observed in the production incident.

### Q-B: env-monitor `EnvironmentNameExact` + Status filter

```sql
DECLARE @env NVARCHAR(64) = 'PROD-NA';
DECLARE @status NVARCHAR(32) = 'Running';

SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT TOP (50) [d].[Id], [d].[Environment], [d].[Status], ...
FROM [deploy].[DeploymentRequest] AS [d]
WHERE [d].[Environment] = @env
  AND [d].[Status] = @status
ORDER BY [d].[Id] DESC;

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

**Purpose:** verifies the HLPS §8 risk that the new index does not regress queries that today seek `IX_Status_IsProd`. The optimiser may pick either index; both are SARGable. Either choice is acceptable provided the plan is a seek (not a scan) and logical reads are low.

### Q-C: main grid AND-path (Project + Environment + BuildNumber distinct values)

```sql
DECLARE @projPrefix NVARCHAR(64) = 'Calc';
DECLARE @envPrefix NVARCHAR(64) = 'PROD';
DECLARE @buildSubstr NVARCHAR(256) = '1.2.3';

SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT TOP (50) [d].[Id], ...
FROM [deploy].[DeploymentRequest] AS [d]
WHERE [d].[Project] LIKE @projPrefix + '%'   -- post-fix: prefix
  AND [d].[Environment] LIKE @envPrefix + '%' -- post-fix: prefix
  AND [d].[BuildNumber] LIKE '%' + @buildSubstr + '%' -- BuildNumber retains substring
ORDER BY [d].[Id] DESC;

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

**Expected post-fix:** index seek on `IX_DeploymentRequest_Project` (most selective predicate typically) followed by residual filtering for the BuildNumber substring on the seek-narrowed row set. Logical reads dominated by the size of the prefix-matched slice.

**Pre-fix:** scan on the clustered index, all three predicates as residuals.

### Q-D: env-monitor OR-path (shared `detailsFilter`)

```sql
DECLARE @details NVARCHAR(256) = 'rabbitmq';

SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- AND with EnvironmentNameExact, OR-combination of Project StartsWith and BuildNumber Contains.
SELECT TOP (50) [d].[Id], ...
FROM [deploy].[DeploymentRequest] AS [d]
WHERE [d].[Environment] = 'PROD-NA'
  AND ( [d].[Project] LIKE @details + '%'
     OR [d].[BuildNumber] LIKE '%' + @details + '%' )
ORDER BY [d].[Id] DESC;

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
```

**Expected post-fix:** seek on `IX_DeploymentRequest_Environment`, then OR predicates evaluated as residuals on the per-environment slice. For a typical environment with 10K rows: a few hundred logical reads.

**Pre-fix:** scan; `LIKE '%@p%'` on Environment + LIKE on Project (substring) + LIKE on BuildNumber (substring), all as residuals.

## 4. Pass criteria

For each query Q-A through Q-D, post-fix must satisfy:

1. **Plan shape:** index seek (not scan) on the new index where applicable. For Q-B, either `IX_DeploymentRequest_Environment` or `IX_Status_IsProd` is acceptable; for Q-C, `IX_DeploymentRequest_Project` (or `IX_DeploymentRequest_Environment`, whichever is more selective in the seed); for Q-D, `IX_DeploymentRequest_Environment`.
2. **Logical reads:** materially lower than pre-fix. The HLPS SC3 ceiling is "low five-figure or lower" for the env-monitor case; the JIT-Spec author / DBA may agree a tighter target after reviewing the seed cardinality. Suggested ceiling for the env-monitor query at production scale: ≤ 5,000 logical reads.
3. **Plan stability for `IX_Status_IsProd`:** queries that today seek `IX_Status_IsProd` must still seek it (or seek a strictly cheaper index) post-fix. No table scan where there was previously a seek.
4. **No new warnings or unexpected plans:** key lookup spikes, hash spills, or sort warnings on otherwise-simple seeks indicate a tuning issue and should be flagged for follow-up.
